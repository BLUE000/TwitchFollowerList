# 基本設計 (Basic Design)

## 1. アーキテクチャ思想
以前のデバイス制御プロジェクトの設計思想を踏襲し、役割ごとにディレクトリ（レイヤー）を分割する。
1. **UI層 (`ui/`)**: 画面の描画やユーザー入力を担当する。
2. **API層 (`api/`)**: 外部との通信（Twitch API、OAuth認証）や、ローカルファイルシステムへのアクセスなど、外部リソースとの入出力を担当する。
3. **Core層 (`core/`)**: UIとAPIの間に立ち、両者をつなぐ内部処理（ビジネスロジックやデータ加工）をメインで担うコントローラー層。

## 2. モジュール構成
1. **UIモジュール (`ui/`)**
   - Qt Designerファイル (.ui) およびそれを制御するViewクラス (`MainWindow` など)。
   - ユーザー操作（ボタン押下など）を受け取り、Coreへ処理を委譲する。データの描画のみに専念する。
2. **Coreモジュール (`core/`)**
   - アプリケーションのメインロジック（Controller / Manager）。
   - UIからの要求を受け、必要に応じてAPIモジュールを利用してTwitch APIを叩いたりファイルシステムを操作し、結果をUIに返す。
3. **APIモジュール (`api/`)**
   - **認証 (Auth)**: Twitch OAuth認証フローの実行、トークンの保存・管理。ブラウザからのリダイレクトを受け取るためのローカルサーバー（`QTcpServer` 等）を一時的に立ち上げ、自動で連携する仕組みを提供する。
   - **通信 (Network)**: Twitch APIへのリクエスト、JSONレスポンスのパース。
   - **ファイル操作 (IO)**: `out` フォルダ内のファイル走査や監視。暗号化処理には外部ライブラリ **TransCipher** を使用する。

## 3. データ連携フロー
1. 起動時、`core` が `api` にトークン検証を依頼。結果を `ui` に伝えてログインボタンの活性・非活性を切り替える。
2. `core` が `api` にファイル一覧の取得を依頼し、結果を `ui` のツリービューに反映する。
3. ログイン完了のタイミングで、`core` が自動的に `api` にAPI通信を依頼してフォロワー一覧を取得する。その後、`core` はデータを `ui` のリストビューに表示させると共に、**TransCipher** による暗号化を施して `out/AllList.dat` として保存させる。

## 4. ディレクトリ構成
```text
FollowerList/
├── doc/            # ドキュメント類
├── out/            # 出力データ配置用ディレクトリ
├── ui/             # UI層: フォームファイル (.ui) と UI制御クラス
├── core/           # 内部処理層: UIとAPIをつなぐアプリケーションロジック
└── api/            # 外部連携層: API通信、OAuth認証、ファイル操作など
```
